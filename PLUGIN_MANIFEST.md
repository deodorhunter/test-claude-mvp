# Golden Example Kit — Plugin Roadmap

> This document tracks the path from "git template" to "org plugin".
> When ready to package, the `.claude-plugin/` directory will be the plugin root.
> Reference architecture: NeoLab CEK (`github.com/NeoLabHQ/context-engineering-kit`).

---

## What Is (and Isn't) in the Plugin

The plugin contains everything that is **org-generic** — identical across all 40 projects.
Project-specific pieces stay in the repo and are generated/customised per project.

```
PLUGIN (org-generic — same for all projects)     PROJECT (per-repo — customised)
─────────────────────────────────────────────    ──────────────────────────────
.claude/commands/                                docs/AI_REFERENCE.md       ← /init-ai-reference
  compress-state.md                              docs/ARCHITECTURE_STATE.md ← scaffolded
  handoff.md                                     docs/backlog/               ← project US
  init-ai-reference.md                           .claude/rules/project/      ← /reflexion output
  judge.md                                       .claude/agents/             ← domain customised
  reflexion.md                                     (file domain paths differ per project)

.claude/rules/org/                               CLAUDE.md
  (promoted from project rules)                    ## Active Project Rules section
                                                   → imports project + org rules

AI_PLAYBOOK.md                                   docs/plan.md
CLAUDE.md (framework sections only,              docs/handoffs/
  not "Active Project Rules")
```

## Phase 1 — Git Template (now, in use)

**How to adopt:**
```bash
# Option A: fork
git clone https://github.com/org/golden-example-kit new-project
cd new-project && /init-ai-reference   # generates docs/AI_REFERENCE.md

# Option B: copy .claude/ directory
cp -r golden-example/.claude new-project/.claude
cp golden-example/CLAUDE.md golden-example/AI_PLAYBOOK.md new-project/
cd new-project && /init-ai-reference
```

**What teams customise:**
1. Agent `owns/forbidden` paths in YAML frontmatter (their file structure differs)
2. `docs/AI_REFERENCE.md` (auto-generated by `/init-ai-reference`)
3. `docs/backlog/BACKLOG.md` (their US, phases, status)

**Limitation:** updates to the framework don't propagate automatically. Teams must manually pull changes.

---

## Phase 2 — Git Submodule for Commands (medium term)

Keep `.claude/commands/` as a shared git submodule. Updates to commands propagate to all 40 repos via `git submodule update`.

```bash
# In each org project
git submodule add https://github.com/org/golden-example-kit .claude/commands
```

**What this gives:** automatic command updates. Teams still customise agents and rules locally.

**Limitation:** submodule UX is awkward. Agent files are still manually copied.

## Phase 3 — Plugin Package (future)

Structure follows NeoLab `.claude-plugin/` convention:

```
golden-example-kit/           ← root of the plugin repo
  .claude-plugin/
    manifest.yml              ← plugin metadata
    commands/                 ← same as our .claude/commands/
    agents/                   ← same as our .claude/agents/ (generic versions)
    rules/
      org/                    ← org-level rules
  CLAUDE.md                   ← framework sections only (no project-specific content)
  AI_PLAYBOOK.md
  README.md
```

**`manifest.yml` (target format):**
```yaml
name: golden-example-kit
version: "1.0.0"
description: "Token-optimized AI development framework. Dual-speed workflow (Copilot + Orchestrator), YAML-frontmatter agents, commands library, rules architecture, and plugin roadmap."
author: org-name
license: internal
compatible_with:
  - claude-code >= 2.1
  - cursor >= 1.0
  - opencode >= 1.0
commands_dir: commands/
agents_dir: agents/
rules_dir: rules/org/
entry_point: CLAUDE.md
project_overrides:
  - docs/AI_REFERENCE.md    # generated per project
  - .claude/rules/project/  # discovered per project
  - .claude/agents/         # domain paths customised per project
```

**Installation (target):**
```bash
/plugin marketplace add org/golden-example-kit
/init-ai-reference   # generates project-specific AI_REFERENCE.md
```

## Rule Promotion Flow

As project rules prove themselves across multiple projects, they graduate to org scope:

```
.claude/rules/project/rule-NNN.md    ← discovered in Project 1
       │
       │ proven in 2+ projects
       ▼
.claude/rules/org/rule-NNN.md        ← shared across all projects (Phase 2)
       │
       │ packaged
       ▼
golden-example-kit/rules/org/        ← in the plugin (Phase 3)
```

**Governance:** rule promotion requires a PR to the `golden-example-kit` repo. The org's tech leads review and approve. This prevents rule bloat at org level.

## Skills Architecture (future)

As teams build project-specific capabilities beyond the base kit:

```
.claude/skills/                       ← project-specific skills (agentskills.io spec)
  analyze-query-performance/
    SKILL.md                          ← loaded when "query performance" is triggered
    examples.md
  setup-tenant-fixtures/
    SKILL.md
    scripts/setup_tenant.py
```

Org-wide skills → `golden-example-kit/skills/` in the plugin.

## Token Budget of the Full Plugin (target)

When the plugin is loaded, what gets injected into context?

| Component | Load timing | Target size |
|---|---|---|
| Plugin manifest metadata | Always | < 500 tokens |
| CLAUDE.md (framework sections) | Always | < 3,000 tokens |
| Active project rules (via `@` imports) | Always | < 1,000 tokens (max 5 rules × 200 tokens) |
| Agent file | On delegation | < 800 tokens each |
| Command file | On trigger | < 600 tokens each |
| Skill file | On trigger | < 500 tokens each |
| **Total always-loaded** | | **< 4,500 tokens** |

Contrast with a naïve "dump everything in CLAUDE.md" approach: 15,000–30,000 tokens always loaded.
