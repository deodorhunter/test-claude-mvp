# US-017: Token Quota Tracking + Rate Limiting (Redis)

**Agente:** Backend Dev
**Fase:** 2d
**Dipendenze:** US-012
**Priorità:** high
**Stato:** 📋 Backlog

## Contesto

Due layer di enforcement:
1. **Redis** (rate limiting, finestra 60s, sub-millisecondo): previene abuse a breve termine
2. **PostgreSQL** (quota mensile, durabile): impone limite totale per tenant

Entrambi controllati PRIMA di ogni model call. Redis key format: `rate:{tenant_id}:{user_id}`.

## Task

Espandere `backend/app/quota/` con implementazione completa:
- `RateLimiter`: Redis sliding window (60s), `check_rate_limit(tenant_id, user_id)` → bool
- `QuotaService`: legge/aggiorna `tenant_token_quota` in DB, `check_quota(tenant_id, tokens_estimated)` → bool, `consume_quota(tenant_id, tokens_used)`
- `QuotaMiddleware` o dependency FastAPI che compone i due check
- Alert quando tenant supera 80% quota (log warning + audit entry)

## Acceptance Criteria

- [ ] `RateLimiter` blocca richieste che superano il limite (default 10 req/min per utente)
- [ ] `QuotaService` verifica quota residua prima di ogni model call
- [ ] `consume_quota()` aggiorna `tokens_used` in DB dopo ogni model call (anche in caso di errore parziale)
- [ ] Rate limit key scoped a `tenant_id:user_id` (no cross-tenant sharing)
- [ ] Alert log quando quota > 80%
- [ ] Audit entry per `quota_exceeded`
- [ ] Unit test: rate limit hit, quota exceeded, consumption tracking, Redis failure handling
- [ ] Completion summary in `docs/progress/US-017-done.md`
