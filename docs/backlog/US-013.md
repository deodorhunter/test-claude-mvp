# US-013: Cost-Aware Planner + Multi-Model Fallback

**Agente:** AI/ML Engineer
**Fase:** 2b
**Dipendenze:** US-012
**Priorità:** critical
**Stato:** ✅ Done

## Contesto

Il planner riceve una request (prompt + contesto) e decide quale modello usare, rispettando:
- Il budget token del tenant (da `tenant_token_quota` in DB)
- La modalità configurata (`AI_MODE`)
- La disponibilità del modello primario (fallback automatico)

**MVP fallback logic:** se Ollama non risponde (timeout 5s) → prova Claude (se API key configurata) → errore 503.

## Task

Implementare `Planner` in `ai/planner/planner.py`:
- `plan(prompt, context, tenant_config) → ExecutionPlan`
- `ExecutionPlan`: sequenza di nodi `{type: model|tool, adapter, estimated_tokens}`
- Stima token pre-esecuzione (approssimazione: len(prompt+context)/4)
- Controlla quota residua prima di selezionare modello (legge da DB tramite dependency injection)
- Fallback: se primario fallisce o quota superata → seleziona alternativa
- Audit log entry per ogni decisione di pianificazione

## Acceptance Criteria

- [ ] `Planner.plan()` ritorna un `ExecutionPlan` con il modello selezionato
- [ ] Se tenant ha 0 token residui → errore 429 (quota exceeded), audit log entry
- [ ] Se modello primario non risponde (timeout 5s) → fallback automatico
- [ ] `ExecutionPlan` include `model_used`, `estimated_tokens`, `fallback: bool`
- [ ] Unit test: happy path, quota exceeded, fallback triggered, modello non disponibile
- [ ] Nessuna chiamata a model API reale nei test (mock)
- [ ] Completion summary in `docs/progress/US-013-done.md`
