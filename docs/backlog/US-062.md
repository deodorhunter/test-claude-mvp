# US-062: Plone-MCP Architecture Clarification — Rename/Document 3 Touchpoints

**Dependencies:** —
**Priority:** medium
**Status:** 📋 Backlog

## Context

User feedback: "why plone-mcp with plone adapter? plone mcp should be used to do things on dockerized plone/volto site or am I misunderstanding something?"

There are 3 separate Plone-related components with confusing naming overlap:
1. `PloneIdentityAdapter` (`backend/app/auth/plone_bridge.py`) — JWT auth bridge validating Plone tokens
2. `infra/plone-mcp/` — MCP server for content CRUD operations on a running Plone/Volto instance
3. `plugins/plone_integration/` — placeholder plugin for future Plone content sync

The user is right that plone-mcp should operate directly on the Plone site. The confusion is that the auth adapter has "Plone" in its name too, suggesting a relationship that doesn't exist at the MCP level.

## Task

1. **Add** clear module docstring to `backend/app/auth/plone_bridge.py`:
   - "This is the JWT auth bridge — it validates Plone-issued tokens, NOT an MCP adapter"
2. **Update** `infra/plone-mcp/README.md`:
   - Clear purpose statement: "MCP server that operates on a running Plone/Volto site"
   - Distinguish from auth adapter explicitly
   - Document supported operations (content CRUD, search, etc.)
3. **Update** `docs/ARCHITECTURE_STATE.md`:
   - Add "Plone Integration Points" section with diagram showing all 3 touchpoints
   - Label each with purpose and data flow direction
4. **Update** `PLUGIN_MANIFEST.md`:
   - Clarify `plone_integration` plugin status (placeholder vs implemented)

## Acceptance Criteria

- [ ] All 3 Plone touchpoints documented with clear distinct purposes
- [ ] `plone_bridge.py` has docstring clarifying it's auth-only
- [ ] `plone-mcp/README.md` explicitly distinguishes from auth adapter

- [ ] User reading the docs can understand which component does what
