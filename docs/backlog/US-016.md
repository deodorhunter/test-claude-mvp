# US-016: RAG Pipeline — Qdrant Embeddings + Retrieval

**Agente:** AI/ML Engineer
**Fase:** 2c
**Dipendenze:** US-014
**Priorità:** high
**Stato:** 📋 Backlog

## Contesto

Il sistema usa Qdrant come vector database per RAG. Ogni tenant ha una collection dedicata (`tenant_{tenant_id}`). Il `tenant_id` arriva SEMPRE dal JWT verificato, mai dal request body. LlamaIndex gestisce embedding e retrieval.

**Modello embedding MVP:** `nomic-embed-text` via Ollama (già nel container), nessuna API key esterna.

## Task

Implementare il RAG pipeline in `ai/rag/`:
- `QdrantStore`: wrapper per operazioni Qdrant con isolamento per tenant
  - `index_document(tenant_id, doc_id, text, metadata)` → crea/aggiorna vettori
  - `search(tenant_id, query, top_k=5)` → ritorna chunks rilevanti con score
- `EmbeddingService`: genera embedding via Ollama (`nomic-embed-text`)
- `RAGPipeline`: orchestra embedding + retrieval + formattazione risultati

## Acceptance Criteria

- [ ] `QdrantStore` usa collection `tenant_{tenant_id}` — mai cross-tenant access
- [ ] `search()` filtra SOLO sulla collection del tenant corrente
- [ ] Embedding generato via `nomic-embed-text` tramite Ollama
- [ ] `RAGPipeline.query(tenant_id, query)` ritorna lista di chunks con score e source
- [ ] Documento indicizzato è recuperabile con query semanticamente simile
- [ ] Unit test con Qdrant mockato (nessuna connessione reale nei test)
- [ ] Tenant isolation verificata: document di tenant A non appare in search di tenant B
- [ ] Completion summary in `docs/progress/US-016-done.md`
