# US-072: Fix Cost Tracking — Dollar Estimates in SESSION_COSTS.md + Drift Detection Recalibration

**Priority:** medium
**Status:** 📋 Backlog

## Context

Two measurement problems were identified in the Phase 3 benchmark run:

### Problem 1: SESSION_COSTS.md tracks tokens, not dollars

All rows in SESSION_COSTS.md report token counts ("~273k", "~454k") without distinguishing token types. Cache read tokens cost $0.30/1M; raw input costs $3.00/1M; output costs $15.00/1M. A session that looks expensive in token count (many cache reads) is actually cheap. The Phase 3d session cost $11.41 in actual API spend; SESSION_COSTS.md implied ~$50–80 based on the token counts.

### Problem 2: Drift detection metric is a false positive

`benchmark/capture-session-metrics.sh` counts drift signals as `toolUseResult` entries with serialized content shorter than 100 characters. In the first real run, 95.2% of tool results triggered this — but investigation showed they were all legitimate: tool metadata objects (`{"filenames":[],...}`), skill results (`{"success":true,...}`), and one user rejection.

The Budget Cap truncation bug (from the ArkNill analysis) manifests as `truncated: true` in Glob and Read tool responses — not as short content length. The current metric is noise.

3. **Update** `benchmark/capture-session-metrics.sh` drift signal counting:

   - Also count: `toolUseResult` entries where `type == "string"` and length < 50 (genuine plain-text truncation)
   - Update the `note` field in output JSON to reflect the corrected methodology

4. **Update** `benchmark/README.md` "Drift signal detection" subsection to document the corrected approach and explain why content length was a false positive.
